# Backend Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development or superpowers:executing-plans. Steps use checkbox (`- [ ]`) syntax.

**Goal:** A Node/TypeScript REST API that manages `ai.fixmsg.properties` as a Git-backed, branchable config file, records who-did-what in an append-only audit log, identifies users by Windows ID, and enforces the merge gate (ERROR blocks, WARNING needs acknowledgement) using the Plan 2 analysis engine.

**Architecture:** Express app built from injectable dependencies (`ConfigRepo`, `UserDirectory`, `AuditLog`) for testability. The Git service shells out to the real `git` CLI over a managed non-bare repo, serializing write/merge operations through an in-process queue. Persistence for users/audit is a dependency-free JSON store (swappable for a DB later). Identity comes from a trusted reverse-proxy header, with a dev-mode fallback.

**Tech Stack:** Node 20+, TypeScript, Express 4, Vitest + Supertest. Imports `@config-manager/rule-engine`.

## API surface (all under `/api`, identity required)
- `GET /me` → current user.
- `GET /users`, `POST /users/:id/role` (admin only).
- `GET /file` → main content. `GET /branches` → names. `POST /branches` {name}.
- `GET /branches/:name/file`, `PUT /branches/:name/file` {content, message}.
- `GET /branches/:name/diff` → unified diff vs main.
- `GET /branches/:name/analysis` → findings + gate summary.
- `POST /branches/:name/merge` {acknowledgeWarnings?, override?, overrideReason?} → enforces gate, then 3-way merge.
- `GET /history` → commit graph.

## Merge gate (POST merge)
1. `evaluateGate(readFile(branch))`.
2. If `errorCount > 0`: block (403) unless `override && user.role==='admin' && overrideReason` (audit records the reason).
3. Else if `warningCount > 0 && !acknowledgeWarnings`: 409 (client must confirm).
4. Merge; on conflict return 409 with conflicted files; on success audit `merge`.

## Audit events (append-only)
`create-branch`, `edit`, `merge` (with `override`/`overrideReason` when used), each with `windowsId`, `ip`, `timestamp`, `branch`, `commit`.

## Conservative/safety choices
- All write/merge git ops serialized (single working tree, no races).
- `core.autocrlf=false` in the managed repo so content round-trips byte-exact.
- Branch names validated `^[A-Za-z0-9._/-]+$` (no leading `-`).
- Reads use `git show ref:file` (no checkout) so concurrent reads are safe.
- A request with no resolvable Windows ID is rejected 401 (never anonymous).
